# 测试驱动开发与测试策略

## 核心铁律

**没有先写失败测试，绝不写生产代码。** 每段代码必经两件事：亲眼看到测试失败（证明它能捕获 bug），再亲眼看到测试通过（证明代码满足需求）。

## 测试策略分级

按**影响半径、共享程度、回归风险、测试价值**四维判定测试等级。

### 分流判定表

| 级别 | 适用场景 | 策略 |
|------|---------|------|
| **Level 0** | 配置调整、文案修改、单行修复、局部小改动 | 定向验证——改完跑相关用例即可 |
| **Level 1** | 中小修复、局部行为变更、边界清晰的单模块改动 | 回归测试——补 1-2 个关键路径测试 |
| **Level 2** | 新功能、明确行为变更、共享逻辑、高风险改动 | TDD——完整红-绿-重构循环 |
| **Level 3** | 跨模块变更、公共 API 修改、并发/持久化相关 | TDD + Code Review |
| **Level 4** | 以上任一完成时 | Completion Verification——全量回归 + 交付闸门 |

判定口诀：**看影响半径。** 改函数内部实现，只影响自己 → Level 0/1。改公共接口签名，影响所有调用方 → Level 2 起步。涉及共享数据结构/持久化 → Level 3。

### Level 0：定向验证

```
改完 → 跑受影响的测试 → 通过 → 完成
```
- 不用写新测试，但必须跑已有相关用例
- 如果没覆盖，手写一行验证脚本，确认后即删
- 禁止顺手重构、顺手加测试、顺手优化

### Level 1：回归测试

```
补 1-2 个关键路径测试（看到失败） → 改代码 → 跑全量 → 通过 → 完成
```
- 测试覆盖修改点的主路径和一个边界
- 目的是防止回退，不是驱动设计
- 不需要严格红-绿-重构的三段式，但测试必须在改代码前写好并**看到失败**

### Level 2+：TDD 完整流程

判定为 Level 2 及以上时，进入以下完整 TDD 流程。Level 3/4 叠加 Code Review 和全量回归。

## 整体流程
```
分流判定 → 准备阶段 → 红-绿-重构循环（逐检查项迭代）→ 收尾全量回归
```
---
## 一、准备阶段

### 1.1 任务拆解

需求明确后，将工作拆解为**可独立验证的检查项**，使用 `todo` 工具创建清单。每个检查项必须**可判断通过/失败**，而非模糊描述。

<Good>
```
检查项示例：
- 空邮箱地址返回验证错误
- 有效邮箱格式通过验证
- 邮箱含特殊字符（+、.）正确处理
```
</Good>

<Bad>
```
- 实现邮箱验证  ← 太模糊，不可判断
- 完善错误处理   ← 无明确验收标准
```
</Bad>

### 1.2 搭建测试框架

若项目尚无测试框架，搭建最小可用框架——确认能运行空测试并**失败**，且不过度构建。

### 1.3 数据与接口设计
任务引入新数据结构或公共接口时，先设计后实现：
- 定义结构、签名和约束
- 为输入、输出和边界情况编写中文文档
- 确认接口变更与用户对齐

### 关键原则：垂直切片，拒绝水平切片
**禁止**一次性写完所有测试再写实现。应垂直切片：
```
WRONG: test1-5 → impl1-5
RIGHT: test1→impl1, test2→impl2, ...
```
---

## 二、红-绿-重构循环

对每个检查项，严格执行以下循环：

### 2.1 红灯——写失败测试

只针对当前检查项编写测试：
- 先写正常路径，再写边界和异常
- 测试应通过公共接口验证**行为**，而非实现细节
- 测试名应像规格说明：「用户使用有效购物车可完成结账」

<Good>
```python
def test_retries_failed_operation_3_times():
    """验证失败操作会重试最多 3 次"""
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise RuntimeError("fail")
        return "success"
    result = retry_operation(operation)
    assert result == "success"
    assert attempts == 3
```
—— 清晰命名，验证真实行为，只测一件事
</Good>

**确认红灯：** 运行测试并亲眼看到它因**正确的原因**失败（功能缺失）。

- 测试通过？→ 你在测已存在的行为，修正测试
- 测试报错？→ 修正错误，重跑直到正确失败

### 2.2 绿灯——最小实现

编写**刚好足够**让测试通过的代码：不添加测试未覆盖的功能，不为未来预留抽象（YAGNI）。

<Good>
```python
def retry_operation(fn, max_retries=3):
    for i in range(max_retries):
        try:
            return fn()
        except Exception:
            if i == max_retries - 1:
                raise
```
—— 刚好够通过测试
</Good>

**确认绿灯：** 运行测试确认通过，且**所有已有测试保持绿色**。

### 2.3 重构——改善设计

只在全绿状态下重构：消除重复、改善命名、提取公共逻辑。保持测试全绿。

**绝对禁止在红灯时重构。**

### 2.4 循环

取下一个检查项，回到「红灯」，重复直至全部完成。

---

## 三、收尾阶段

### 3.1 全量回归

全部检查项通过后，运行**全部测试套件**，确保未破坏既有逻辑。

---

## 反模式与红线
以下任一情况发生 → **停，删代码，重来：**

- 代码写在测试之前
- 测试写完后立即通过（没看到红灯）
- 无法解释为什么测试会失败
- 「我先手动测过了」
- 「这次跳过 TDD 更快」
- 「补测试也能达到同样效果」
- 「已经花了几小时，删掉太浪费」

**测试困难 = 设计有问题。不要绕过困难——改善设计。**
