# 先验研究

## 核心原则

你遇到的问题，大概率已有人解决过。动手前先把用户描述映射到已有术语和成熟方案。

## 写作风格

- 语气直接、克制。像同事写内部 wiki
- 禁止套路过渡语：不说「众所周知」「不言而喻」「值得注意的是」
- 禁止逢迎语气：不说「希望对你有帮助」「欢迎留言讨论」
- 禁止贴标签式概括：不说「外壳 vs 灵魂」「两层声音」这类 slogan
- 允许「注意」「换句话说」「对比来看」等引导语，克制使用
- 引用优于转述——给出原始出处让读者自行验证

## 输出格式

直接生成 HTML 方案卡片，写到当前工作目录，命名 `YYYY-MM-DD-标题.html`。

HTML 规范：
- 完全自包含，不依赖任何 CDN 或外部资源
- 字体使用系统栈，不加载 web font
- 设计服务于阅读，不为炫技。无渐变、无 box-shadow 堆叠、无 backdrop-filter
- 响应式：移动端可读
- 内容保持方案对比表 + 结论结构

## 工作流

### 1. 术语映射

把用户描述翻译成业界通用术语，找不到时用多个候选词并行搜索：

- 「数据流水线」→ ETL / data pipeline / workflow engine
- 「实时消息推送」→ WebSocket / SSE / long polling
- 「代码生成器」→ codegen / AST transformation / scaffolding
- 「分布式锁」→ distributed lock / consensus algorithm

### 2. 解决方案调研

对每个术语，搜索三层信息：
- **概念** —— 百科、技术博客、论文，`web_search`
- **实现** —— 开源库、框架、SaaS，`web_search` + `read`
- **权衡** —— 竞品对比、社区讨论，`web_search`（加 `vs`/`comparison`）

### 3. 生成 HTML 方案卡片

按以下结构生成 HTML，关键信息标注来源：

- **术语映射** —— 用户描述 → 标准术语
- **已有方案** —— 方案对比表（方案、成熟度、适用场景、不适用原因）
- **结论** —— 推荐方案及理由

### 4. 决策分流

- 成熟方案直接满足 → 推荐使用，进入集成设计
- 部分满足需定制 → 明确差异，以成熟方案为基础扩展
- 无现有方案 → 罕见。说明搜索覆盖范围，再进入设计

## 质量自检

- [ ] 术语映射覆盖了用户描述的所有关键点
- [ ] 每个候选方案给出了成熟度和适用场景
- [ ] 不推荐方案说明了具体原因，而非笼统否定
- [ ] 结论有明确推荐理由
- [ ] 搜索范围充分，未遗漏主流方案

## 反模式

- 「太简单不用调研」→ 简单轮子被重复发明最多
- 「调研浪费时间」→ 自研到半才发现已有方案才是真浪费
- 「已有方案太重」→ 确认重在哪，能否只用子集
- 「学原理所以自研」→ 可声明学习目的，别包装成生产方案
