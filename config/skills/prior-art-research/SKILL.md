# 先验研究——不重复造轮子

## 核心原则
你遇到的问题，大概率已有人解决过。动手前先把用户描述映射到已有术语和成熟方案。

## 何时触发
用户表达以下意图时，须先执行本标准流程：

- 提出「新」算法/架构/协议
- 从零搭建工具/框架/DSL
- 描述概念但未用通用术语
- 「我有个想法」但无已有工作引用
- 自己实现已有成熟库能做的事

**例外（可直接动手）：**
- 已引用已有方案并说明为何不用
- 明确「我知道 X，但要做 Y 差异」
- 纯练习/学习目的（明确声明）

## 工作流
### 1. 术语映射
把用户描述翻译成业界通用术语，找不到时用多个候选词并行搜索：
```
「数据流水线」    → ETL / data pipeline / workflow engine
「实时消息推送」  → WebSocket / SSE / long polling
「代码生成器」    → codegen / AST transformation / scaffolding
「分布式锁」      → distributed lock / consensus algorithm
```
### 2. 解决方案调研
对每个术语，搜索三层信息：

| 层级 | 内容 | 工具 |
|------|------|------|
| **概念** | 百科、技术博客、论文 | `web_search` |
| **实现** | 开源库、框架、SaaS | `web_search` + `read` |
| **权衡** | 竞品对比、社区讨论 | `web_search`（加 `vs`/`comparison`） |

### 3. 输出调研结论
```markdown
## 术语映射
- 用户描述 → 标准术语 A / B / C
## 已有方案
| 方案 | 成熟度 | 适用场景 | 不适用原因（如有） |
|------|--------|---------|------------------|
| X    | 成熟   | ...     | ...              |
## 结论
- 可直接使用 X（推荐，理由）
- 可组合 A + B 实现（理由）
- 需要自研，但与已有方案的核心差异是 Y
```

### 4. 决策分流
- **成熟方案直接满足** → 推荐使用，进入集成设计
- **部分满足需定制** → 明确差异，以成熟方案为基础扩展
- **无现有方案** → 罕见。说明搜索覆盖范围，再进入设计

## 反模式
- 「太简单不用调研」→ 简单轮子被重复发明最多
- 「调研浪费时间」→ 自研到半才发现已有方案才是真浪费
- 「已有方案太重」→ 确认重在哪，能否只用子集
- 「学原理所以自研」→ 可声明学习目的，别包装成生产方案
