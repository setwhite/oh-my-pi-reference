# 文章写作

## 核心准则

### 写作风格
基础规则遵循 `prose-style` skill，以下为补充。

- 面向有基础的技术读者，假定读者能读懂代码。不解释基础语法，不科普行业常识
- 完全禁止emoji——正文、标题中一律不用
- 禁止感叹号。禁止反问句

### 联网搜索规则
必须搜索的情况：
- 引用官方文档、RFC、规范——验证准确性和时效性
- 涉及性能数据、benchmark——查证来源和测试条件
- 涉及版本差异、API变更——核实版本号和变更记录
- 涉及第三方库的使用方式——查官方 README 或源码

可以不搜索的情况：
- 纯逻辑推演或算法分析
- 用户已提供完整上下文且仅需要组织表达
- 个人实践经验总结（需标注「基于个人经验」）

搜索结果矛盾时，优先采信官方文档；官方文档过时则采信源码。

### 内容密度
- 正文每段 ≤ 5 句。超过则拆分或用列表/表格
- 代码块必须有行内注释说明关键步骤，不另建参数说明表
- 三段文字能说清的，不加图

### 内容结构
```

标题（一句话说清解决什么问题，≤ 25 字）

│ 撰写日期：YYYY 年 M 月 D 日

一、省流

≤ 3 句。直接给结论或核心观点，不铺垫，不设问。

二、正文

一级到四级标题编号：一、 → （一） → 3. → （4）
以下模块按需组合，不必全部出现：
- 问题区 —— 解决了什么问题、为什么值得写
- 原理区 —— 机制解释，配合代码或图
- 实践区 —— 可复现的步骤、代码示例
- 对比区 —— 方案/工具/版本对比，优先用表格
- 踩坑区 —— 遇到的错误及原因，不罗列显而易见的注意事项

三、总结

≤ 3 句。回顾核心结论，不复述正文小节标题。

参考链接

列出正文中引用的所有外部来源，每条附带链接文本（非裸 URL）。
正文未引用的链接不列在这里。

```

### 文件交付
- 单文件，写到当前工作目录
- 命名：`YYYY-MM-DD-标题.md`（如 `2026-07-18-能指与所指的区别.md`）
- 标题取自主题，中文

## 工作流

### 一、搭骨架
按输出结构出大纲，仅到结构层，不写正文：
```

标题

省流方向

正文模块

- 模块1：类型（问题区/原理区/实践区/对比区），覆盖内容
- 模块2：...
待搜索

- 搜索项1
- 搜索项2
```
大纲发给用户，用户确认后再继续。

### 二、搜索执行
按大纲中的待搜索清单，并行搜索多组关键词。信息不充分时扩大搜索词再搜一轮。
搜索完成后交叉验证关键事实（版本号、API 名称、性能数字）。

### 三、写正文
按确认后的大纲展开。

### 四、冗余自检
生成后回读全文，逐项检查：
- 同一件事说了两遍？合并或删除
- 有感叹号、反问句、emoji？删除
- 有「希望对你有帮助」等社交套话？删除
- 有相邻的短段落可以合并？合并
- 代码块后的文字段落是否在重复解释代码中已有的注释？删除
- 表格内容是否比段落更啰嗦？改用段落
- 所有引用是否在参考链接中可点击？

## 反模式

- 省流写成摘要——省流是结论，不是正文的缩水版
- 正文写成教程——不要「首先…其次…最后…我们来试试…」。给结论，再解释
- 总结写成目录——不要「本文介绍了 A、B、C 三个方面」
- 代码块无注释 + 下方一大段参数说明——代码块本身应自解释
- 搜索了但不引用——搜索结果被丢弃，正文凭记忆写
- 大纲环节就开始写正文——大纲只到结构层
- 加「小结」「延伸思考」「后记」等额外章节——三部分结构够用
