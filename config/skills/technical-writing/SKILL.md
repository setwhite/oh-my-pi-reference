# 技术文章写作

## 核心原则

**技术文章的目的是让读者懂，不是展示你搜到了多少信息。**

每写一句话，问自己：删掉它，读者理解这个主题会不会受影响？不影响就删。

KISS 兜底。不设禁词表和句式规则——信任模型能判断什么是套话，但你得先告诉它什么才是重要的。

---

## 输出结构（严格按此顺序）

```
# 标题（一句话说清这篇文章帮读者解决什么问题）

> **撰写日期**：YYYY 年 M 月 D 日
> **适用版本/时效**：（如不适用可省略此行）

## 省流
（1–2 句话，不加修饰，让读者 3 秒决定要不要继续）

## 目录
（锚点链接，≤ 7 条）

## 正文（每节按以下模板）
### N.1 节标题
**一句话说明**：（这节讲什么）

| 维度 | 说明 |
|------|------|
| ... | ... |

**要点**：
- ≤ 5 条核心信息
- 每条 1–2 句话

**适用/不适用**：（至少写一条不适用场景或注意事项）

---

## 总结
（≤ 3 句话，不引入正文没出现过的新信息）

## 参考链接
（如有）
```

**为什么是这个结构？**

- **日期置顶**：知乎时间戳在页面底部。读者需要立刻判断文章是否过时。
- **省流**：技术读者扫描式阅读。前三秒抓不住就划走了。
- **表格优先于段落**：对比性信息、规格参数、优缺点——表格比段落少占用 50% 的阅读时间。
- **要写不适用场景**：一篇文章只讲优点不讲边界，读者踩坑回来骂你。信任就没了。
- **总结不超过 3 句**：总结的作用是让人记住，不是复读一遍正文。

## 工作流：从调研结论到完稿

### Step 1：确认输入

如果前面已经跑过 prior-art-research，直接用它的输出——不要再搜索。prior-art-research 的「已有方案」表和「结论」就是你的原材料。

如果没有 prior-art-research 的输出，才走简化搜索：
- 使用 `web_search` 工具进行联网搜索，按子问题分批搜索、逐批整理
- 搜索前，把主题拆成 2–3 个子问题
- 每个子问题搜索结果压缩为 ≤ 3 条关键事实 + 来源
- 去重归类

### Step 2：搭骨架——大纲先行

按输出结构出大纲，**先不写正文**。大纲里定好：
- 每节的标题和目的（这节帮读者解决什么）
- 哪些信息用表格、哪些用要点
- 每节预估要点数量

把大纲发给用户确认。**不要跳过这一步直接写全文。** 骨架偏了，再好的文字也是废的。

### Step 3：写正文

按大纲逐节展开。每节写完自查：
- 表格是不是确实比段落更适合这里的信息？
- 要点有没有超过 5 条？超过 5 条说明这节需要拆分。
- 有没有写不适用场景？
- 每段手机屏能不能一眼看完？（≤ 4 句话）

### Step 4：篇幅控制

如果全文超过目标字数（默认 3000–5000 字），不要手动逐句删——做**砍半**：

1. 从每节挑出最核心的 50% 要点保留
2. 能合并成表格的段落合并
3. 检查有没有同一个意思用了两句话在说——有就删一句

删完后重新读一遍「省流」和「总结」，确保它们和砍半后的正文一致。

## 质量自检（输出前逐条确认）

```
[ ] 日期是否在文章最顶部？
[ ] 是否有「省流」？是否 ≤ 2 句话？
[ ] 目录是否 ≤ 7 条？
[ ] 每节是否有表格或要点（不是纯段落）？
[ ] 每节是否写了不适用场景/注意事项？
[ ] 总结是否 ≤ 3 句话？是否引入了新信息？
[ ] 对比性信息是否优先用了表格？
[ ] 每段手机屏能否一眼看完？
[ ] 读一遍全文：有没有删掉也不影响理解的句子？有就删。
```

## 反模式

- 「本文将详细介绍...」→ 读者知道你在介绍，不用你说。直接开始。
- 正文纯段落、无表格、无要点 → 技术读者不读长段落，他们扫读。
- 每节要点超过 5 条 → 不是节太长就是你取舍不够狠。
- 总结超过 3 句，或引入了新信息 → 总结不是追加，是收束。
- 写了优点不写边界 → 宣传稿，不是技术文章。
- 先写全文再让用户看 → 骨架偏了等于白写。大纲先行。
- 收到 prior-art-research 输出后又自己搜一遍 → 重复劳动。信上游的输出。
