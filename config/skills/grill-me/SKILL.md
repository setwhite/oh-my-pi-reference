# Grill Me — 盘问我，直到需求清晰

## 核心原则

模糊的需求产出的代码，删掉重写的概率是清晰需求的 10 倍——宁可多花 5 分钟盘问，也不要花 5 小时改错。

## 何时触发

- 用户描述存在歧义（一种说法可对应多种实现）
- 存在多套技术上可行的方案，且各方案有明显不同的取舍
- 需求缺少关键约束（性能、规模、兼容性、安全级别）
- 用户用了模糊词汇：「优化」「完善」「改进」「弄好」
- 改动可能影响公共接口、共享数据结构、或跨模块边界

**例外（可直接执行）：** 「按你的判断来」「选最保守的方案」；纯局部改动，边界清晰，只有一种合理实现；问题在代码库中已有先例。

## 工作流

### 1. 先探索，后提问

能通过以下方式回答的，不问用户：

1. 读相关代码、配置、文档
2. 查已有实现模式（找先例）
3. 看 git 历史（为什么这么写）

**只有穷尽了代码库能回答的信息后，才向用户提问。**

### 2. 识别决策点

把模糊需求拆成**决策树**，每个分支节点标注「已有信息可判断」或「需用户确认」。

```
├── 缓存粒度？
│   ├── 方法级
│   └── 数据级
├── 失效策略？
│   ├── TTL
│   ├── LRU
│   └── 手动清除
├── 存储后端？
│   ├── 内存（进程内）
│   ├── Redis
│   └── 文件
└── 是否需要分布式？
    ├── 单机即可
    └── 需要共享缓存
```

### 3. 结构化盘问

用 `ask` 工具一次性提问，而非逐条轰炸：

- 一次最多问 3 个问题
- 每个问题给 2–4 个选项，标注推荐项
- 选项描述写清取舍，不只写标签
- 能合并的问题不拆开

### 4. 共识后生成计划

用户确认所有决策点后，生成**结构化计划文件**再编码，包含：边界与范围、关键决策、验证方式。通过 `local://plan.md` 持久化，作为单一事实来源。

## 反模式

- 「我觉得用户是这个意思」→ 你觉得 ≠ 用户觉得。问。
- 「先做出来再让用户看」→ 需求猜错的话，产出全是废代码。
- 「这个问题太小了不值得问」→ 小问题攒成大返工。
- 「我给 5 个方案让用户选」→ 太多选项 = 没有帮助。2–4 个，带推荐。
- 「代码库看不懂，直接问用户吧」→ 先读代码。用户不是你的代码搜索引擎。
