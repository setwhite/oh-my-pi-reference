# 先验研究——不重复造轮子

## 核心原则

**你遇到的问题，大概率已经有人解决过了，而且解决得比你好。** 在动手构建之前，先把用户的想法映射到已有术语和成熟方案上。

这不是限制创造力——是让你站在巨人的肩膀上，而不是在巨人的脚边重新发明轮子。

## 何时触发

用户表达以下意图时，必须先执行本标准流程，再进入设计或实现：

- 提出一个「新」的算法、数据结构、架构方案
- 想从零搭建某个工具、框架、DSL、协议
- 描述一个概念但没有使用业界通用术语
- 说「我有个想法」但未引用任何已有工作
- 试图「自己实现」已有成熟库能做的事

**例外（可直接动手）：**
- 用户明确引用了已有方案并说明为什么不用
- 用户说「我知道有 X，但我要做的是 Y 差异」
- 纯粹的练习/学习目的（明确声明）

## 工作流

### 1. 术语映射

把用户的自然语言描述翻译成**业界通用术语**：

```
用户说的              → 搜索确定的术语
「数据流水线」         → ETL / data pipeline / workflow engine
「实时消息推送」       → WebSocket / SSE / long polling
「代码生成器」         → codegen / AST transformation / scaffolding
「分布式锁」           → distributed lock / consensus algorithm
```

如果找不到确切术语，用多个候选词并行搜索。

### 2. 解决方案调研

对每个映射到的术语，搜索三层信息：

| 层级 | 搜索内容 | 工具 |
|------|---------|------|
| **概念** | 维基百科、技术博客、论文摘要 | `web_search` |
| **实现** | 开源库、框架、SaaS 产品 | `web_search` + `read` 仓库/文档 |
| **权衡** | 竞品对比、Hacker News 讨论、Reddit 帖子 | `web_search`（加 `vs`、`comparison`、`alternative`） |

### 3. 输出调研结论

用以下结构汇报：

```markdown
## 术语映射
- 用户描述 → 标准术语 A / B / C

## 已有方案
| 方案 | 成熟度 | 适用场景 | 不适用原因（如有） |
|------|--------|---------|------------------|
| X    | 成熟   | ...     | ...              |

## 结论
- 可直接使用 X（推荐，理由）
- 可组合 A + B 实现（理由）
- 确实需要自研，但与已有方案的核心差异是 Y
```

### 4. 决策分流

调研结论决定下一步：

- **已有成熟方案直接满足需求** → 推荐使用，进入集成方案设计
- **部分满足，需定制** → 明确差异点，以成熟方案为基础扩展
- **确实无现有方案** → 这是罕见情况。说明搜索过程和覆盖范围，再进入设计

## 反模式

- 「这个太简单了，不需要调研」→ 简单的轮子被重复发明得最多
- 「调研浪费时间」→ 自研到一半发现已有成熟方案，才是真浪费
- 「已有方案太重了」→ 先确认重在哪里，是否可以只用子集。不要凭印象判断
- 「我想学习原理所以自己实现」→ 学习目的明确声明则可，但不要包装成生产方案
