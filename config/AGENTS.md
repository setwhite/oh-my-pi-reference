# Python 规范

WHEN 需要安装或管理 Python 包时，
    DO 使用 uv
    DON'T 使用 pip

WHEN 编写 Python 代码时，
    DO 严格遵循 PEP8
    DON'T 违反编码风格约定

WHEN 编写 Python 函数或声明变量时，
    DO 写出完整的类型提示（Type Hints）
    DON'T 依赖隐式推断

# 核心准则

WHEN 编写代码和撰写文档时，
    DO 遵循 KISS 原则，保持极简，只解决当前需求
    DON'T 过度设计、编写投机性内容，或添加未要求的灵活性

WHEN 用户提出新想法、新方法或新概念时，
    DO 先调用 `web_search` 对标已有领域的术语、理论和成熟实践，再判断其新意所在
    DON'T 将已有成熟东西重新包装成"原创框架"、以及重复造轮子

WHEN 遇到需求模糊或存在多套方案时，
    DO 调用 `ask` 工具向用户提问，并说明假设与权衡
    DON'T 盲目猜测、自行决定，或盲目编写代码和文档

WHEN 编写注释、Docstrings、设计文档或提交信息时，
    DO 强制使用中文，以降低理解门槛与认知负荷
    DON'T 使用任何非中文语言撰写内容（专业术语中不可避免的外文除外）

# 代码工作流（TDD）

step 0 — 任务拆解：
WHEN 接到新任务时，
    DO 明确成功标准，将任务拆解为多个步骤，并使用 `todo` 工具创建清单，清单项格式为 `步骤 -> 验证: [检查项]`
    DON'T 直接开始编码

step 1 — 数据建模：
WHEN 任务目标与验证清单明确后，
    DO 定义数据形态（名称、字段、关系、约束），用稳定的数据模型消除复杂控制流
    DON'T 跳过数据设计直接编写代码

step 2 — 接口设计：
WHEN 数据模型设计完成后，
    DO 设计模块或函数接口，编写中文 Docstrings，明确输入、输出及边界条件，保持概念完整性。若项目尚无测试框架，此时搭建
    DON'T 忽略接口定义直接实现

step 3 — 红灯：编写测试
WHEN 接口设计完成后，
    DO 取一个检查项编写测试——先正常路径，再边界和异常。运行并确认红灯
    DON'T 跳过测试直接实现，或一次写完所有测试

step 4 — 绿灯：最小实现
WHEN 测试红灯后，
    DO 编写刚好足够的代码使测试通过
    DON'T 实现测试未覆盖的功能，或为将来预留代码

step 5 — 循环
WHEN 测试绿灯后，
    DO 回到 step 3 取下一个检查项，重复红-绿循环直至全部通过
    DON'T 检查项全部通过前进入回归

step 6 — 全量回归
WHEN 全部检查项通过后，
    DO 运行全部测试套件，确保未破坏既有逻辑
    DON'T 跳过全量回归

# 代码维护

WHEN 修改现有代码时，
    DO 确保每一行 Diff 都直接追溯到当前需求
    DON'T 附带无关的优化、注释或格式修改、重构未损坏的逻辑

WHEN 既有代码基础上工作时，
    DO 强制沿用现有代码风格
    DON'T 擅自更改代码风格

WHEN 清理本次改动产生的死代码时，
    DO 只删除因本次改动而废弃的依赖、变量或函数
    DON'T 擅自删除历史遗留的死代码；如发现，可以提示用户，但严禁自行删除
